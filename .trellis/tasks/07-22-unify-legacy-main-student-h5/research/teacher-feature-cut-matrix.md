## Evidence and implications

- The current question workbench already supports editing, publishing, and rejecting candidates or drafts, with textbook evidence and duplicate-risk gates that legacy lacks. A wholesale legacy question-page port would regress the current RAG-backed workflow.
- The current teacher console already places global smart-assessment configuration in System Settings and class overrides in Class Settings. Legacy's separate paper page would create two owners for the same policy.
- Current teacher-account management lives in the platform-operations application and has stricter dependency and password-reset behavior than legacy. Its approved replacement is a supervisor-only section inside current Teacher Settings; teacher self-password change remains available to everyone.
- Latest legacy deliberately collapses product roles to `teacher` and `student`. Any authenticated teacher can list teacher accounts, create a teacher with an initial password, and enable or disable another teacher from the same Settings page. The unified product adopts its placement and small action set but restricts peer management to the existing privileged/supervisor capability.
- Legacy account management intentionally omits account deletion and role editing, which keeps the embedded workflow small and avoids a second administrator product. It also lacks an other-teacher password-reset action.
- The legacy `must_change_password` flag is stored and displayed, but `LegacyTeacherApp` does not gate a teacher with that flag into a forced password-change screen. The unified implementation must enforce the flag if the product continues to promise “首次登录必须修改密码”.
- Disabling an account is safer than deleting it because authentication checks account status on subsequent requests and historical ownership remains intact. Main's stronger password reset/versioning and session-revocation behavior can be retained without retaining the operations frontend.
- The analytics taxonomy is confirmed: latest legacy's element-family mapping is primary, while current experiment, point, attempt, report, and evidence details remain the drill-down implementation.
- Legacy-only commits provide useful regression cases even when their UI is not ported, notably roster import password validation, explicit student-ID conflicts, safe class deletion semantics, final catalog placement deletion, question draft lifecycle, and report-dialog overflow.
